## 1. 사용자 가정: 기본값은 비개발자

이 저장소는 비개발자가 LLM의 도움을 받아 직접 코드를 고치고 배포하는 것을 전제로 합니다.
사용자가 명시적으로 "개발자입니다", "제가 직접 코드를 보겠습니다"처럼 밝히기 전까지는 항상 비개발자 모드로 응답하세요.

### 비개발자 모드

- 항상 친근한 한국어로 자세히 설명하세요.
- 무슨 파일을 왜 보는지, 무엇을 바꾸는지, 결과가 어떤 의미인지 풀어서 말하세요.
- 어려운 용어는 사람 말로 바꿔 설명하세요.
  - `ENOENT: no such file` → "필요한 파일이 없어요. 제가 찾아서 만들게요."
  - 타입 에러 → "값 형식이 안 맞는 부분이 있어요. 제가 고칠게요."
  - 빌드 실패 → "사이트로 만들기 전에 검사 단계에서 막혔어요. 원인을 찾아 고치겠습니다."
- 사용자가 직접 명령어를 치게 하지 말고, LLM이 가능한 범위에서 실행하세요.
- 단, 파괴적 명령은 반드시 사용자 동의를 받고 실행하세요.
- 한 단계가 끝날 때마다 "지금까지 한 것 / 다음에 할 것"을 한두 줄로 요약하세요.
- 최종 답변에는 결과, 확인 방법, 남은 위험을 비개발자가 이해할 수 있게 자세히 적으세요.

### 개발자 모드

- 사용자가 본인을 개발자라고 밝히면 평소 LLM 톤으로 간결하게 진행해도 됩니다.
- 그래도 파괴적 명령 규칙과 Git 확인 규칙은 그대로 지키세요.

## 3. 작업 시작 전 Git 확인

코드 변경, 기능 추가, 버그 수정, 문서 수정, 배포 작업을 부탁받으면 손대기 전에 반드시 실행하세요.

```bash
git fetch origin
git status
```

결과에 따라 이렇게 대응합니다.

- 원격에 새 커밋이 있는 경우:
  > "원격(GitHub)에 새로 올라온 변경이 있어요. 충돌을 막기 위해 먼저 `git pull --rebase origin main`부터 진행해도 될까요?"

- 로컬에 안 커밋된 변경이 있는 경우:
  > "지금 저장은 안 했지만 수정 중인 파일들이 있어요: [파일 목록]. 이걸 마저 정리하고 작업할까요, 아니면 그대로 위에 이어서 작업할까요?"

- 깨끗한 상태:
  > "Git 상태가 깨끗해서 바로 작업을 시작하겠습니다."

읽기 전용 질문에서는 생략할 수 있지만, 세션 시작 시에는 그래도 Git 정리 제안을 먼저 하는 것을 기본으로 합니다.

## 6. 사용자가 "OK / 배포 / 반영해줘"라고 답한 경우

다음 순서대로 자동 실행하세요.

1. `git status`로 변경 사항 확인
2. 관련 파일만 `git add`
3. `git commit -m "<한국어 요약>"`
4. `git push origin main`
5. 배포 진행 중 안내

```text
🚀 GitHub에 올렸어요! 배포가 진행되고 있어요.
1~2분 정도 걸려요. 다 끝나면 알려드릴게요.
```

6. `gh run watch`가 가능하면 사용, 안 되면 `gh run list --branch main --limit 1`로 확인
7. 배포 완료 안내

```text
🎉 배포 완료!

사이트가 새 버전으로 바뀌었어요:
👉 https://choe1535-alt.github.io/estimate-coffee/

브라우저에서 새로고침(Cmd+Shift+R)을 한 번 해주시면 확실히 보여요.
```
